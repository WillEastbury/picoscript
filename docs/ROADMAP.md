# PicoScript platform roadmap

Tracked feature work for the PicoScript platform (reference toolchain +
`docs/playground.html` UI + the JS/C# runtimes in `baremetaljstools` and
`developercli`). Status legend: ☐ planned · ◐ in progress · ☑ done.

Foundation already shipped:
- ☑ **Workflow frontend** — `picoscript_workflow.py` (`--lang workflow`): visual
  step-list JSON → natural-English → PicoIL. See
  [docs/WORKFLOW_DIALECT.md](WORKFLOW_DIALECT.md). Byte-aligned with
  `BareMetal.WorkflowPico` (JS) and the C# `PicoVm` oracle.
- ☑ English `For each X in <collection>:` value iteration.

---

## 1. Workflow as a first-class designer / code surface

Elevate **workflow** to a first-class surface in `docs/playground.html`
(generated by `gen_playground.py`) alongside C / BASIC / Python / English /
COBOL / report / functional — with a **visual designer** as its primary editor.

- ☐ **Register** `workflow` in `ALL_STYLES` + `COMPILERS` (`gen_playground.py`),
  and make it compile/run in-browser (workflow → English via WorkflowPico →
  existing in-browser English compiler).
- ☐ **Visual designer** — embed a node/step designer (reuse
  `BareMetal.Workflow.designer`): drag/add/edit/reorder steps ⇄ JSON step list.
- ☐ **Derived views** — English, PicoIL, bytecode hex, and VM run output panes;
  surface the `warnings[]` list for host-only steps.
- ☐ **Cross-language translate** — workflow → English → {C, BASIC, Python} views.
- ☐ **Examples** — arithmetic+conditional, array `FOREACH` sum, reject/budget
  flow in the playground catalog.

## 2. Report designer

A first-class **report designer** built on the existing `picoscript_report.py`
(4GL) frontend: visual band/column report → report model → PicoIL, rendered via
the `Json` / `TextRender` / `Utf8Writer` hooks.

- ☑ **Report model schema** — data source (array / variable-in-Memory), detail row
  expression, `where` filter, and `count`/`sum`/`min`/`max` aggregates
  (`picoscript_reportmodel.py`).
- ☑ **Compiler** — `compile_report_model(json) → PicoIL` (lowers to English:
  materialise rows in `Memory`, loop the detail band, print the aggregate footer);
  wired as `--lang report-model` (`.rptmodel`) in `picoscript_build.py`. Tested in
  `tests/test_report_model.py`.
- ☐ **Designer UI** — visual band/column designer surface in the playground
  emitting the report model JSON, with a live rendered preview.
- ☐ **Richer rendering** — Storage/Query data sources, text/column formatting via
  `TextRender` / `Utf8Writer` / `Json`, grouping; C# VM golden differential.

## 3. Event hooks — RAISE & subscribe

First-class **RAISE** and **SUBSCRIBE / ON** event hooks across the frontends and
runtimes. Builds on existing primitives:
- `RAISE` opcode `0xE` (`Thread.Raise` / swirq)
- the `Event.*` reactive queue `0x0180–0x0186` (`Post` / `Next` / `Type` /
  `Target` / `Data` / `SetData` / `Count`)
- the `ON Ns.Method:` `OnBlock` AST node and `Raise` node
- the `CAP_EVENT` capability

Work items:
- ☑ **RAISE surface** — `RAISE`/`EMIT` workflow step → `Event.Post`; `Event.*` is a
  uniform host-call primitive in every frontend.
- ☑ **Subscribe surface** — `ON`/`SUBSCRIBE` workflow block → an `Event.*`
  drain-and-dispatch loop; handler runs per matching pending event.
- ☑ **JS runtime** — `BareMetal.Workflow` engine bridges `RAISE`→`PubSub.emit` and
  `ON`→`PubSub.on`.
- ☑ **C# runtime** — the `Event.*` FIFO (Post/Next/Count/Type/Target, 0x180–0x186)
  in `WorkflowHost`; `oracle.js` `wf_events` case reproduced by the C# `PicoVm`.
- ☑ **Workflow steps** — `RAISE`/`ON` in all three workflow compilers
  (`BareMetal.Workflow`, `BareMetal.WorkflowPico`, `picoscript_workflow.py`).
- ☑ **Tests & docs** — reference/browser/cross-language tests; see
  [EVENTS.md](EVENTS.md).
