# PicoScript synthetic dataset

Training data for fine-tuning a small coding model to write **PicoScript** — built
and **verified** by the real toolchain, so it cannot contain an invalid example.

Generated by [`tools/gen_dataset.py`](../tools/gen_dataset.py); validated by
[`tests/test_dataset.py`](../tests/test_dataset.py).

## How it's built (and why it's trustworthy)

Each program is assembled from a dialect-agnostic statement model, rendered into
all four surfaces (C-style / BASIC / Python-style / natural-English), then **every
rendering is compiled and run** on PicoVM. A program is emitted **only if**:

1. all four dialects compile, **and**
2. all four run to the **same output**, **and**
3. Python/BASIC/English are **byte-for-byte identical** bytecode (shared AST + lowerer).

So cross-dialect equivalence is a *free, machine-checked correctness label* — there
is no hand-labelling and no unverified code in the corpus.

Constructs covered: arithmetic, modulo, conditionals, `for`/`while`/**nested** loops,
factorial, power, even-sum, **ternary**, **switch / jump-table dispatch**.

## Files

| file | format | tracked? |
|------|--------|----------|
| `picoscript.chat.sample.jsonl` | small chat-format sample (browse the shape) | ✅ committed |
| `picoscript.jsonl` | raw records: `nl2code` / `translate` / `run` | ⛔ gitignored (regenerable) |
| `train.chat.jsonl` / `val.chat.jsonl` | OpenAI-style chat messages, train/val split | ⛔ gitignored (regenerable) |

**Chat record** (the fine-tune format):

```json
{"messages":[
  {"role":"system","content":"You write PicoScript … in the natural-English dialect …"},
  {"role":"user","content":"Sum the integers from 1 to 8 and print the total."},
  {"role":"assistant","content":"Set s to 0.\nFor each i from 1 to 8:\n    Increase s by i.\nPrint s."}],
 "meta":{"task":"nl2code","dialect":"english","construct":"t_sum_range"}}
```

Two task types are emitted as chat: **`nl2code`** (instruction → code, per dialect)
and **`translate`** (dialect A → dialect B, exact-behaviour). The raw `run` records
(code → output) are kept for an execution-prediction signal but not exported to chat.

## Regenerate

```bash
python tools/gen_dataset.py --count 400 --seed 1 --val-frac 0.1
# -> data/{picoscript.jsonl, train.chat.jsonl, val.chat.jsonl, picoscript.chat.sample.jsonl}
```

A `--count 400` run yields ~400 verified programs → ~2000 chat examples
(~1800 train / ~200 val). Increase `--count` for more; diversity is bounded by the
template parameter space (add templates in `gen_dataset.py` to grow it).

## Fine-tune notes

- The chat files drop straight into most SFT pipelines (the `messages` schema is
  the common chat format). For a base/BitNet model without a chat template, flatten
  `system+user → prompt`, `assistant → completion`.
- **Verification as eval:** the strongest metric for a PicoScript model is *does the
  generated program compile and run to the expected output?* — reuse the same
  `compile + run` check this generator uses. Cross-dialect agreement is a bonus
  consistency metric.
